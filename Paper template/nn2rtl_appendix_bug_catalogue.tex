\subsection{Per-module verification evidence state}

ResNet-50: all 119 modules register as passing in the pipeline state, and 117 of 119 carry fresh on-disk evidence, of which 108 are bit-identical and 9 are within the three-unit tolerance. The remaining 2 modules lack complete fresh result files because later integration changes altered the testing environment and their original per-module passes were not rerun. Their final per-module evidence is therefore incomplete; nevertheless, the assembled ResNet-50 backbone passes the stricter byte-exact network-level gate.

MobileNetV2: the recorded System-1 cost snapshot covers 97 of the 99 modules, of which 82 are bit-identical and 15 are within tolerance. The global-average-pool and classifier modules were added afterwards, and the complete network passes the assembled network gate (Section~\ref{sec:rq3}). The per-module bit-identical and within-tolerance split for those two deferred modules is not separately recorded.

A representative module-level check was particularly extensive: layer1\_0\_conv1 was verified bit-identical, with a maximum error of 0 and a first-mismatch index of $-1$, across 6,422,528 samples. This is the depth of a single well-characterised check and should not be read as evidence that every module was sampled to the same extent.

\subsection{Per-class gate mapping}

\begin{enumerate}
\item Stale-artifact and regeneration omission: surfaces at the assembled network gate; localised by comparing against a freshly regenerated self-consistent golden, checking artifact modification times, and running the full ordered regeneration chain before trusting any end-to-end figure.
\item Missing or wrong activation-rescale arithmetic: surfaces at the assembled network gate; localised by a multi-step un-confounded bracket (a byte-exact $1\times1$ control, a Python recompute that is position-exact against the RTL, and goldin triangulation) and by fitting the fixed-point transfer over the entire input domain rather than a single vector.
\item Operand half-swap, scaling and tiling layout: surfaces at the assembled network gate; localised by an operand probe on the exact transfer handshake (both inputs byte-exact while the output is wrong) followed by offline replay of on-disk goldens in both operand orientations.
\item Handshake, backpressure and deadlock: surfaces at the assembled network gate; localised by live simulation-state inspection of scheduler, bridge and loader registers, by the freeze signature (ready\_out=1 with valid\_out=0), and by a self-consistency test that separates real datapath errors from handshake artifacts.
\item Latency and memory-read-latency contract: splits across gates; pipeline-latency violations are caught at the per-module gate, while memory-read-latency violations surface only once integrated and are localised by an engine-isolation harness with controllable weight-read latency.
\item Simulation-artifact false-positive: a verification-side class; resolved by running Verilator with X-initialisation forced to zero (the FPGA-faithful condition) and single-threaded, and by inspecting mapped primitive counts rather than the functional readback.
\item Contract, bus-width and packing: surfaces at the assembled network gate; localised by reading RTL port widths against the contract map and by coherent multi-stage patching, since adjacent stages are mutually tuned to each other's lossy contract.
\item Engine datapath arithmetic and counter-leak: caught by the strict zero-mismatch assembled network gate, since high overall byte-exactness otherwise hides it; localised by a single-pixel-isolation testbench, a per-pixel mismatch grid, and a drop-which-MAC elimination model.
\item Component byte-exact but end-to-end wrong: the assembled network mismatch is treated as the primary functional oracle in simulation; localised by a per-node probe bisection downstream of the first wrong node and by Python triangulation that recomputes a layer from its goldin, weights and scale.
\item Golden-reference float-accumulation artifact: a verification-side class in which the RTL matches the intended integer arithmetic while the floating-point-based golden introduces the discrepancy; localised by trying two scale choices, where a persistent identical residual implicates the golden rather than the RTL.
\item Autonomous-pipeline pattern-reuse and toolchain hazards: a pipeline-side class; caught by deterministic pre-flight and structural gates, by separating toolchain-infrastructure failures from RTL failures, and by gating cache promotion on the final verification and Vivado/PPA gate.
\item Frontend and accuracy-measurement: a measurement-side class; resolved by measuring through the deployment integer path or with batch-norm set to identity, and by confirming that the scale ratio equals the per-channel batch-norm fold factor.
\end{enumerate}

\subsection{Extended investigation: B21}

During end-to-end verification of ResNet-50, a sparse error affecting roughly 2.7\% of outputs appeared in wide-channel residual additions in the network's later stages, present only in the composed chain; every implicated module had passed its per-module gate in isolation. Localising it required an eight-hypothesis adversarial code review across 83 handshake-fed nodes. The review first attributed the error to a spatial\_run handshake asymmetry and applied a fix that gated valid\_in by spatial\_run across those nodes, recorded as B21. That attribution was later superseded: the true defect was a missing per-layer activation requantisation in 22 of 48 ReLU modules (B7), where the template had emitted a plain \(\max(0,x)\). The B21 handshake change was retained only because it independently reduced cycle count by about 12\%. The episode shows that, even after the assembled network gate flags a composed-design failure, root-cause attribution can be hard and a plausible but incorrect fix can mask the true cause, so the byte-exact comparison rather than any single hypothesis remained the arbiter.

\section{Shared Compute Engine}\label{app:engine}

The shared time-multiplexed engine, like all of the generated hardware, was produced by the integration agent (System~2) and was not specified by the human: it comprises a 256-lane signed multiply-accumulate array feeding 256 wide accumulators, a three-stage requantisation pipeline, an address generator, a configuration-register block, and a memory-to-stream bridge. Its structure was the agent's own; the author set goals and approved milestones but supplied no wiring diagram or module specification.

\section{Per-Module Failure Classification and Pattern Reuse}\label{app:selfimprove}

The per-module pipeline classifies each generation failure into one of sixteen recurring arithmetic and structural error classes (for example integer overflow, missing saturation, or unfolded batch-normalisation), augmented by four infrastructure classes that separate genuine RTL defects from toolchain problems. This failure-memory and pattern-reuse cache retains the broken RTL and the full failure history, so later modules on related layers begin from accumulated experience. When a module passes, it also stores a pattern note with reference RTL; a note is promoted to active guidance after it has helped three later modules and archived if a module that later uses it fails, with promotion gated on the module still passing the final verification and Vivado checks. This is a reuse mechanism rather than a measured self-optimisation.
