# Static Verilator Testbench

Handwritten infrastructure. Never regenerated by an agent.

## Files

- `static_verilator_tb.cpp` — the shared driver runtime. Contract templates in `contracts/<contract_name>/testbench.cpp` include this runtime after declaring the selected contract.
- `third_party/json.hpp` — [nlohmann/json](https://github.com/nlohmann/json) v3.11.3, single-header JSON library. Vendored to keep the C++ build self-contained.

## Build contract

The testbench is compiled once per module. Two preprocessor macros select the DUT:

```
-DVMODEL_HEADER="\"V<module_id>.h\""   (generated Verilator header)
-DVMODEL_CLASS=V<module_id>            (generated Verilator class name)
```

`run_verilator` sets these when it invokes `verilator --cc --exe --build -CFLAGS ...`, copies the selected `testbench_template_path` to the build directory, then executes the produced binary with the sidecar path and validates the structured results JSON it writes.

## Canonical signal names

The testbench accesses DUT ports by name: `clk`, `rst_n`, `valid_in`, `ready_in`, `valid_out`, `data_in`, `data_out`. Any generated module must use exactly these names.

Handshake semantics:

- `valid_in` / `data_in` — driven by the bench.
- `ready_in` — **output** of the DUT (backpressure upstream). The bench waits for `ready_in == 1` before asserting `valid_in`. DUTs that do not need backpressure must tie `ready_in` high after reset.
- `valid_out` / `data_out` — outputs of the DUT. The bench samples `data_out` only on cycles where `valid_out == 1`, tolerating bubbles between outputs.

The sidecar carries all seven signal names for documentation and is validated at run time — a module that declares any different name fails before simulation starts with a clear error.

This convention is reflected in Foundry's system prompt and in the `LayerIR` spec used by Cartographer.

## Sidecar schema

Passed to the testbench as `argv[1]`. Assayer generates it. Shape:

```json
{
  "module_name": "conv1x1_block_1",
  "module_id": "block_1_conv1",
  "clock_signal": "clk",
  "reset_signal": "rst_n",
  "valid_in_signal": "valid_in",
  "valid_out_signal": "valid_out",
  "ready_in_signal": "ready_in",
  "data_in_signal": "data_in",
  "data_out_signal": "data_out",
  "bus_bytes_per_sample": 1,
  "input_width_bits": 8,
  "output_width_bits": 8,
  "pipeline_latency_cycles": 3,
  "clock_period_ns": 20.0,
  "golden_inputs_path": "/abs/path/to/block_1_conv1.goldin",
  "golden_outputs_path": "/abs/path/to/block_1_conv1.goldout",
  "results_path": "/abs/path/to/block_1_conv1.results.json",
  "testbench_template_path": "/abs/path/to/contracts/flat-bus/testbench.cpp",
  "contract_id": "flat-bus",
  "contract_name": "Flat Bus",
  "contract_metadata_path": "/abs/path/to/contracts/flat-bus/metadata.json",
  "beat_width_bits": 8,
  "beats_per_input_sample": 1,
  "beats_per_output_sample": 1,
  "weights_path": "/abs/path/to/block_1_conv1_weights.hex",
  "weight_bank_paths": [],
  "axi_weight_data_width_bits": 64,
  "contract_params": {}
}
```

`bus_bytes_per_sample` is the explicit input bus width in bytes and must equal `input_width_bits / 8`.
Contract-specific fields are optional for old sidecars; new Assayer sidecars include them so the selected template and metadata can be audited. For `dram-backed-weights`, `weights_path` is required and the testbench uses it to service the AXI weight read channel.

## DRAM-backed AXI weight model

When `contract_id` is `dram-backed-weights`, the runtime drives a deterministic AXI read responder:

- accepts `weights_arvalid` with `weights_arready` when no burst is active
- treats `weights_araddr` as a byte address into `weights_path`
- treats `weights_arlen` as AXI burst length minus one
- drives `weights_rvalid`, `weights_rdata`, and `weights_rlast`
- advances beats only when the DUT asserts `weights_rready`

The weight file is parsed as a `$readmemh`-style INT8 hex byte stream. Each 64-bit read beat returns eight little-endian bytes.

The golden input/output files use the binary NN2V v2 format:

- `magic`: ASCII `NN2V`
- `version`: uint32 LE = `2`
- `num_vectors`: uint32 LE
- `samples_per_vector`: uint32 LE
- `bytes_per_sample`: uint32 LE
- payload: `num_vectors * samples_per_vector * ceil(bytes_per_sample / 4)` little-endian int32 words

Each logical sample is a packed bus value for one cycle. The bench uses `bytes_per_sample` from the header when unpacking channels and compares results per int8 channel, not per packed word.

## Results schema

Written to `results_path`. Shape matches the `VerifResult` contract minus the fields Assayer derives itself:

```json
{
  "module_id": "block_1_conv1",
  "status": "pass" | "fail",
  "expected": [int, ...],
  "got": [int, ...],
  "max_error": int,
  "mean_error": float,
  "timing_pass": bool,
  "timing_actual_cycles": int,
  "timing_expected_cycles": int,
  "failure_class": null,
  "verilator_stderr": "",
  "axi_weight_memory_model_enabled": true,
  "axi_weight_ar_handshakes": 1,
  "axi_weight_r_beats": 1,
  "axi_weight_completed_bursts": 1
}
```

`failure_class` is always `null` from the testbench. Assayer classifies failures after reading the results file.

The numerical pass threshold is `max_error <= 3` (from the README). The timing pass requires `timing_actual_cycles == pipeline_latency_cycles` exactly.
