# 2. System & methodology

nn2rtl is an LLM-assisted multi-agent pipeline that autonomously generates verified, on-chip-only (no external DRAM) synthesizable RTL for full CNNs on AMD FPGAs (source: memory/project_thesis_source_map.md; memory/MEMORY.md). Three structural layers (source: README.md lines 39-46, 208-249; ARCHITECTURE.md lines 68-289):

- **(1) Claude Code plugin** (`nn2rtl-plugin/`) — defines agent roles + skills.
- **(2) Agent SDK orchestrator** (`sdk/`) — the deterministic TypeScript control plane.
- **(3) Local MCP server** (`mcp/`) — exposes synthesis/verification tools to agents.

Both systems are summarised once more for attribution (source: memory/pipeline-current-status.md (Phase 0-2) + project framing): **system (1)** = the autonomous per-module pipeline; **system (2)** = the integration agent — a separate Claude Opus 4.8 Code-agent instance that built the engine, wired/patched the top wrapper, and ran all debugging + Vivado via multi-agent workflows (the human wrote no RTL and ran no commands).

[DONE] [board: n-a] Autonomous-pipeline summary: 3 LLM agents (cartographer=sonnet-4-6, foundry=opus-4-7, surgeon=opus-4-7), a TypeScript orchestrator on @anthropic-ai/claude-agent-sdk, and a local MCP server with 6 tools (run_iverilog/verilator/vivado, read_weights, write_verilog, get_rtl_patterns); verification is deterministic (no LLM Assayer); weights are never passed to LLMs (.hex + $readmemh) — value: 3 agents / 6 MCP tools (source: docs/THESIS_SOURCE_MAP.md:40-41,24,39).

## 2.3 Orchestration order & feedback edges

[DONE] [board: n-a] **Flow:** start orchestrator -> if no layer_ir.json invoke Cartographer -> init pipeline state (all modules pending) -> loop tick(): invoke_foundry (Foundry then deterministic Assayer) or invoke_surgeon (load broken module, Surgeon, then Assayer) -> classify any failure -> applyVerifResult -> on pass run Vivado synth-only -> Vivado failure feeds back as synthesis_failed -> on terminal write pipeline_summary.json (source: README.md lines 334-388 (Pipeline Flow + Module State Machine); ARCHITECTURE.md lines 184-201 (runPipeline)). Module state machine: pending -> generating -> verifying -> pass | fail_retry (-> generating) | fail_abort. Authoritative record: `output/pipeline_state.json`.

[DONE] [board: n-a] **Feedback edges / self-improve:** the failure-classifier verdict gates retry policy (code_bug -> retry budget; architectural_fit & unknown -> fail_abort); a retry-exhausted code_bug or architectural_fit -> Retrospector advisory -> one final Foundry attempt via SDK resume; passing modules in self-improve mode write probationary docs (promoted to active after 3 successful users, archived on failure); the contract walk falls back flat-bus -> tiled-streaming -> dram-backed on exhaustion. Value: doc promotion threshold default 3 (NN2RTL_DOC_PROMOTION_SUCCESSES); self_improve default ON (NN2RTL_SELF_IMPROVE) (source: README.md lines 354-367, 554-562; sdk/config.ts lines 87-104; ARCHITECTURE.md lines 196-223; knowledge/IMPLEMENTATION_PLAN.md lines 5-33). ## 2.5 Deterministic bit-exact Verilator verification loop

[DONE] [board: n-a] A static handwritten C++ testbench (`tb/static_verilator_tb.cpp`) is the reference checker — never agent-generated (avoids the two-bug problem). ## 2.8 On-chip-only / no-DRAM enforcement (and why URAM is excluded)

[DONE] [board: Alveo U250] **Memory policy is on-chip only, no external DDR** — a supervisor constraint. All weights live in on-chip UltraRAM/BRAM and all activations in BRAM; the dram-backed-weights contract exists but is treated as a separate research artefact, not the deployment mechanism, under this constraint (source: docs/nn2rtl_u250_deployment_plan.md lines 12-13, 42-64, 395-405; docs/nn2rtl_u250_deployment_plan_mobilenetv2.md lines 12-13, 32-50). Per-LayerIR, weights are loaded via $readmemh from `output/weights/*.hex`.

[DONE] [board: Alveo U250] **U250 board memory layout:** 64 GB DDR4 across 4 banks (16 GB each), NO HBM; off-chip weight storage on U250 would need MIG IP + URAM cache — value: 64 GB DDR4 (4 x 16 GB), no HBM (source: memory/pipeline-current-status.md L25). The design is deliberately on-chip-only (no external DRAM).

[DONE] [board: Alveo U250] **Why URAM is excluded from weights:** UltraRAM CANNOT be initialized with non-zero values on the U250 (xcu250 / VU13P), proven on Vivado 2025.2 three independent ways; "weights -> URAM" requires RUNTIME loading, not bitstream $readmemh init — value: ram_style=ultra + $readmemh -> WARNING [Synth 8-10226] falls back to BRAM (0 URAM / 1 RAMB36E2); XPM ultra + MEMORY_INIT_FILE -> [Synth 8-12183] ignored (0 URAM / 8 RAMB36E2); URAM288_BASE primitive has 0 content-init params vs RAMB36E2's 384 INIT_xx params (source: memory/project-uram-no-init.md L12-21; pipeline-current-status.md L26-31). This is a hardware constraint (UG573), not tooling — 2025.2 is latest, fallback exhausted; it forces the design to stay BRAM-bound (weights alone ~5090 BRAM36-equiv far exceed 2688). SUPERSEDES: earlier note "weights->URAM makes the design fit U250" (it fits spatially but requires a runtime load path, NOT bitstream init). A post-impl functional sim reads back correct values ONLY because it fell back to BRAM — a passing readback does NOT prove URAM init works.

[DONE] [board: Alveo U250] **INT4 weight footprint is the binding fit lever:** BRAM bitstream-initializable capacity ~99.1 Mbit (2688 BRAM36); INT4 weights total 93.8 Mbit (spatial 53.4 + engine 40.4) — nibble-packing is the binding fit lever (no DRAM, all on-chip), proven on Vivado 2025.2 (Synth 8-10226/8-12183) (source: docs/agent_tasks/int4_imagenet_timemux_autonomous_plan.md (Known facts, Phase 3 FINDING); newestDate 2026-05-28). SUPERSEDES: engine time-mux gives ZERO BRAM saving (no DRAM, URAM can't init -> all weights resident in BRAM regardless). Runtime buffers (FIFO/act/line-buf ~40 Mbit) map to URAM zero-init and don't compete for BRAM.

### 4.4 RQ3 — integration/correctness bugs & verification

This section is the heart of RQ3: the verification needed to trust LLM-generated hardware. It establishes two cross-cutting claims — that **per-module numerical correctness does not imply end-to-end correctness**, and that **summary statistics (mismatch %, pass/fail) routinely hide outcome-relevant bugs**. The overwhelming majority of catalogued bugs originate from the *integration agent* (the separate, largely-autonomous Claude Opus 4.8 Code-agent instance that built the shared engine, wired the top wrapper, and ran all end-to-end debugging via multi-agent workflows — the human author wrote no RTL and ran no commands), not from the autonomous per-module pipeline. Both are AI; the distinction is which AI system produced the artifact, not human-vs-AI. **Board context:** all simulation-level bugs are tool/Verilator/iverilog-level (board = n-a); the Vivado synthesizability and route observations are on **Alveo U250** unless noted; the early Layer-1 / improve-sweep numbers are **Artix-7** / **ZCU102** as tagged inline.

#### 4.4.2 Category taxonomy (12 categories)

Each category: a one-line definition, member IDs (and count), and the detection methodology that actually works. **Board = n-a (simulation/tooling)** except where the category touches Vivado synthesizability (B72/B74), which is Alveo U250.

1. **Stale-artifact / regeneration-omission** (7: B1, B2, B3, B4, B5, B6, B60) — *the RTL is correct but a derived artifact (packed weights, engine bias/scale map, contract golden, mp_k packing) is left stale after a quantisation regen, silently corrupting e2e while every per-module test passes.* *What works:* compare against a **freshly regenerated, self-consistent golden** and check artifact **mtimes** against the `generate_golden` run; run the **full ordered regen chain** before trusting any e2e number. Component isolation cannot catch these (iso goldens come from the same stale pipeline). Dominant ResNet root cause: B1.
2. **Missing / wrong activation-rescale arithmetic** (2: B7, B8) — *a generation template emits pure `max(0,x)` or a vector-overfit transfer, omitting the per-layer requant when in/out scales differ.* *What works:* a four-step **un-confounded bracket** (byte-exact 1×1 control; Python recompute must equal the RTL position-exact; goldin triangulation; isolate to the ReLU input) — and **fit the fixed-point transfer over the entire input domain**, never one vector.
3. **Operand half-swap, scaling & tiling-layout** (8: B9, B10, B11, B12, B13, B14, B15, B16) — *operands wired into the wrong concat halves, raw scales not divided by out_scale, or golden tiles not interleaved per beat.* *What works:* an **operand probe on the exact transfer handshake** showing both inputs byte-exact yet the output wrong, then an **offline replay of on-disk goldens with the actual RTL params** in both orientations.
4. **Handshake / backpressure / deadlock / lockstep-freeze** (17: B17, B18, B19, B20, B21, B22, B23, B24, B25, B26, B27, B28, B29, B30, B31, B32, B33) — *pulse-style producers ignore downstream ready, asymmetric residual-add joins desync under rate changes, or store-and-drop FIFOs silently lose beats.* *What works:* **live simulation-state inspection** (a purpose-built Verilator probe — written by the integration agent — reading scheduler/bridge/loader registers as ground truth); the freeze signature `ready_out=1 ∧ valid_out=0` localises starvation; an **impossible occupancy ratio** reveals beat duplication. Static count audits produce mostly **false positives**. A **self-consistency test** (a lossless module must emit identical values under varying backpressure) cleanly separates real datapath errors from handshake artifacts.
5. **Latency-contract & memory-read-latency** (4: B34, B35, B36, B51) — *the engine pipeline assumes 1-cycle weight reads while deployed URAM is 2-cycle, or a prefetch `<` should be `<=`, so a MAC multiplies a stale weight.* *What works:* a clean **engine-isolation harness with a controllable weight-read latency** — byte-exact at WLAT=1, exact in-chain error magnitude+count at WLAT=2 — which de-confounds the false "simulator artifact" verdict.
6. **Simulation-artifact false-positives** (4: B37, B38, B39, B40) — *uninitialised-state X-poisoning, multithread-scheduler hazards, undriven wires reading 0 vs X, or a post-impl readback that succeeded only via a silent BRAM fallback.* *What works:* **only Verilator `--x-initial 0`** (models FPGA GSR) is hardware-faithful; **single-threaded** Verilator is the only trustworthy gate; **lint** catches undriven wires; for memory mapping, **inspect the mapped primitive count**, not the readback.
7. **Contract / bus-width / packing mismatches** (6: B41, B42, B43, B44, B45, B46) — *adjacent stages disagree on tiled vs packed-full layout beyond the flat-bus cap, scale widths truncate, or last-bit scale constants diverge.* *What works:* **RTL port-width read against the contract map** (SELRANGE warnings flag width slicing), and **end-to-end coherent multi-stage patching** — these stages are mutually tuned to each other's lossy contract.
8. **Engine datapath arithmetic / counter-leak (high byte-exactness masks a systematic error)** (5: B47, B48, B49, B50, B51) — *a counter advances one cycle too many or a read is gated off by one, dropping exactly one MAC product per pass.* *What works:* a **single-pixel-isolation TB** vs `.goldout`, a **per-pixel mismatch grid**, and a **drop-which-MAC elimination model**; crucially, **map the mismatch count to input-activation sparsity** — the data-dependence (B49) explains why one root cause looks like a 9-layer spectrum and why some layers pass despite the bug.
9. **Component-byte-exact-but-e2e-wrong & verification-method** (9: B52, B53, B54, B55, B56, B57, B58, B59, B61) — *local gates and isolation goldens pass yet the composed design is wrong because the iso goldens were generated consistent with the buggy assumptions, or the standalone checker feeds the wrong input.* *What works:* treat the **e2e mismatch as the oracle** and march a **per-node output-probe bisection** downstream from the first wrong node, with a **known-good control node** held byte-exact; **Python triangulation** (recompute the layer from contract goldin + weights + scale, must equal logical goldout) is the load-bearing de-confounding technique; `equiv_one` is **unreliable for spatial convs**.
10. **Golden-reference float-accumulation artifacts (RTL is more correct)** (2: B62, B63) — *the float32 reference accumulator loses precision the integer RTL keeps, so a 1-LSB "mismatch" is the golden being wrong.* *What works:* try **two different scale choices** — a persistent identical residual implicates the golden; **hardening to integer requant is only safe where the helper provably reproduces the RTL constants over the full domain** (node_mean yes; relu/add no).
11. **Autonomous-pipeline self-improvement & toolchain hazards** (12: B64, B65, B66, B67, B68, B69, B70, B71, B72, B73, B74, B77) — *the learning loop persists state from transient infra failures, switches contracts conceptually-not-executably, caches regressions, loses op coverage, mishandles paths, or hangs.* *What works:* **deterministic preflight/structural gates** (`procedural_declaration_forbidden`, `activation_memory_in_async_reset_block`, `output_counter_missing`, bus-width capability gate), separating `toolchain_infra` from RTL failures, **gating cache promotion on the final Vivado/PPA gate**, and **surfacing raw numeric evidence with no pre-written hypothesis** (B77).
12. **Frontend / accuracy-measurement** (2: B75, B76) — *measuring the deployed network's accuracy is itself an error-prone verification step (BN double-counting), and the HLS baseline cannot even ingest the model.* *What works:* measure through the **deployment integer path** or BN-set-to-identity; confirm the scale ratio equals the per-channel BN fold factor (corr 1.0).

(Category member counts sum to 78 because B51 is listed in both category 5 and category 8 — it is one bug, the address_generator counter-advance, appearing under both the latency-contract and the counter-leak lenses. The catalogue contains 77 distinct bugs B1..B77.)

#### 4.4.3 Four representative bugs (cause / symptom / fix)

**(i) FSM-lock / hardcoded-exit deadlock — B45 (engine FSM exit hardcoded to max OC), with B30 (ST_REQUANT tie-offs).** *Symptom:* the engine FSM enters `ST_REQUANT` and never advances for any layer narrower than the maximum channel count, deadlocking the design. *Root cause:* the `ST_REQUANT` exit condition was hardcoded `oc_pass_idx == (MAX_OC/MAC_COUNT-1) = 2048/256-1 = 7`, but the per-layer oc-pass counter wraps at the layer's actual count (3 for cfg_oc=1024, 1 for 512, 0 for 256) — so for any OC<2048 the comparison never fires (source: `docs/agent_tasks/13a_system_wiring.md`, BLOCKER 2). A sibling FSM-lock (B30) had `requant_valid_in`/`mac_valid_in`/`requant_bias_in` tied to constants *outside* the `ifndef` guard, so the engine sat in `ST_REQUANT` forever even with real sub-blocks. *Fix:* change the exit to `oc_pass_idx == oc_pass_total_m1[2:0]` (board = Alveo U250); remove the 8 tie-offs (skeleton L240-247) and drive the signals from the address generator + bias memory. *Detection:* a second independent integration audit reading the FSM exit against the per-layer counter wrap. **DONE.**

**(ii) `$readmemh` path-failure → all-zero output — B56 (with B55).** *Symptom:* hard all-zero output from the first engine-fed residual add onward; every engine conv computes 0. *Root cause:* the engine weights and bias are preloaded via `$readmemh` from *relative* paths (`output/weights/uram_weights_bank{0..7}.mem`, `bias.mem`), but the Verilator executable ran with its cwd in the build directory, so the files were never found and the URAM/bias memories stayed zero (source: `memory/project_e2e_value_verification.md` L16-17). The cycle-count testbench shared the same defect but, feeding zeros, was unaffected — masking the problem. The sibling B55 is the same hazard in the spatial path: `apply_spatial_scale_path.py` wrote a relative `SCALE_PATH` that did not resolve in the sim cwd, so the scale ROM loaded as zero and all spatial outputs were zero. *Fix:* run the executable with cwd at the repo root (no RTL change) for B56; make `SCALE_PATH` absolute (matching WEIGHTS_PATH/BIAS_PATH) for B55. *Verification:* outputs non-zero (board = Alveo U250). **File-path conventions are a real correctness hazard.** **DONE.**

**(iii) Stride-2 / rate-change desync — B22 (MP-increase residual-add desync), with B29 (MBV2 final-stage).** *Symptom:* raising spatial output-channel parallelism (ResNet MP 16→32, stem 8→16) on 38 convs hard-deadlocked e2e with the output stuck at 0 (no wrong values — a clean wedge). *Root cause:* a rate-sensitive beat-misalignment at an asymmetric residual-add join — the rhs arm is buffered by a depth-512 skid but the lhs arm (conv_202) drains directly into the add with no output skid, so the faster MP=32 producer shifts conv_202's valid_out phase, one beat slips, and the synchronized two-stream join can never pair beats again. conv_202 overproduced (37016, past the 25088 frame bound) while the add wedged 224 beats short (source: `memory/project_mp_increase_deadlock.md` L10-23). The MBV2 analogue B29 is the same desync on the 2-beat lo/hi modules conv_884 (C=576) / conv_908 (C=960) when all 17 depthwise were raised to MP=16. *Fix (proposed, byte-exact):* add a symmetric output skid_fifo (DEPTH=512) on the conv_202→add LHS path; the project kept the byte-exact MP=16 baseline (13,348,787 cyc = 15.0 fps @200MHz, meets the 10 fps target). *Detection:* a cycle-accurate Python FSM model confirmed the datapath byte-exact and terminating at MP=32, and counters refuted the spatial_run hypotheses, localising to the asymmetric add join (board = Alveo U250). **PARTIAL** (root-caused; fix proposed; byte-exact MP=16 retained).

**(iv) Synthesizability anti-pattern: array write in an async-reset block — B72.** *Symptom:* Verilator passes (often bit-exact, `timing_pass=true`) yet the synthesis preflight hard-fails with `activation_memory_in_async_reset_block`, and the failure recurs because the agent treats the rule as contract-specific. *Root cause:* an indexed array write lives inside `always @(posedge clk or negedge rst_n)`; the async-reset edge in the sensitivity list blocks Vivado BRAM/LUTRAM inference even if the reset branch never writes the array — so the memory dissolves into thousands of discrete registers and the module cannot place (source: `knowledge/patterns/protected/08_common_bugs.md`, board = Alveo U250). *Fix:* split the memory write into a dedicated `always @(posedge clk)`-only block, restoring RAM inference — a recurring fix across *every* weight contract, caught by a deterministic structural preflight gate before Vivado is ever invoked. The same synthesizability lever shows up quantitatively in the improve-loop (ZCU102): converting unsynthesisable register-dissolved cones back to inferable form took **node_conv_298 reduce-ff from 267,918 → 6,254 FFs (-97.7%)** and **node_conv_284 reduce-lut from 227,703 → 64,530 LUTs (-71.7%)** (source: `docs/nn2rtl_supervisor_explanation.md` L511-564; the project sentinel `999,999` LUTs = "Vivado could not place at all", e.g. node_conv_298 reduce-lut 999,999 → 89,499 = -91.1%). **DONE.** **A recurring synthesizability anti-pattern across every weight contract.**

#### 4.4.4 The two cross-cutting RQ3 methodology claims

**Claim 1 — component byte-exactness does NOT imply end-to-end byte-exactness.** Concrete instances:
- *Every per-module gate passed, yet 3 audit rounds found **12** cross-piece bugs* (B52). The local gates are structurally blind: port-consistency cannot judge whether a spec *value* is correct (the 16-bit scale_mult, B43), iverilog tolerates floating nets and empty module bodies (the missing bias path B16/B30, the empty FIFO/URAM stubs), and per-sub-block testbenches never co-simulate (the engine FSM tie-offs B30 never met the real sub-block outputs).
- *All ~99 MBV2 modules and all 34 engine dispatches were byte-exact in isolation, yet both tops deadlocked* (B41, B53) — and the dominant error in fact began at the *very first* engine dispatch (B12), the opposite of the "dispatches 0-12 correct" assumption. The paradox resolves because the isolation goldens were generated *consistent with* the buggy integration parameters (operand convention B10, activation layout B12).
- *The engine datapath was byte-exact in isolation at WLAT=1 but produced an exact in-chain error at the deployed WLAT=2* (B34) — the isolation harness inadvertently matched the engine's *wrong* assumption.
- *The standalone equivalence checker (`equiv_one`) falsely fails byte-correct spatial convs* (B54) and falsely passes/fails depending on whether iso goldens are stale (B60), so the only reliable signal is **in-chain e2e against a fresh golden**.

The methodological consequence adopted: a system-level integration-smoke co-simulation gate (engine + scheduler + a tiny dispatch) at the wave-review boundary, and a hard rule that the **e2e mismatch is the oracle**, localised by per-node probe bisection.

**Claim 2 — summary statistics (mismatch %, pass/fail) HIDE outcome-relevant bugs.** Concrete instances:
- A **99.9960% byte-exact** engine layer (**2** of **50,176** bytes wrong) concealed a *systematic* dropped-last-MAC datapath bug (B47) — the headline match rate would have passed any percentage threshold.
- The **same** counter-leak bug (B48, B49) produced 0 mismatches on some layers and up to **8,578** on others purely because the dropped product is `a[ic=0]·w` and vanishes when that activation byte is zero — so a pass/fail gate on the clean layers (conv_250/254/294/300) would have shipped a real bug, and the per-layer mismatch count is a *data-dependent* statistic, not a correctness statistic.
- A **4.1%** divergence between a fresh and a stale final golden (B2) meant every prior "byte-exact" milestone and every fluctuating "2728"/"±1" e2e number was suspect, while a real ~16% error stayed hidden behind the stale comparison baseline (B1).
- A ~94% deep-conv "mismatch" was a pure X-init **simulation artifact** (B37); the hardware-faithful residual was only **2.72%** — the raw percentage sent debugging down a multi-day false RTL-bug path. A **688**-byte "mismatch" was a multithread-scheduler artifact (B38).
- A persistent **2/8000** "mismatch" was the **golden** being less correct than the integer RTL (B62) — here the non-zero statistic flagged a *golden* bug, not an RTL bug.

### Safely open-sourceable vs gated
**Open-sourceable (deterministic, in-repo):**
- The autonomous pipeline (orchestrator sdk/orchestrate.ts, agents Cartographer/Foundry/Surgeon, Failure-Classifier/Retrospector, deterministic Assayer) and the generated RTL.

- The protected knowledge library: 12 hand-written pattern docs (01_context..12_depthwise_conv), 4 protected reference Verilog files, 13 generated lifecycle docs, 3 reference .v files (conv1x1/conv3x3/conv7x7 passing_reference.v) written in-repo under LICENSES.md (source: docs/nn2rtl_supervisor_explanation.md lines 286-315; knowledge/references/protected/LICENSES.md).

**Gated / not bitstream-portable:**
- **Vivado**: serialize all synth (one MobileNetV2 depthwise OOC ballooned vivado.exe to ~75 GB; 4 parallel OOC drove a 100 GB machine to 95% RAM), RAM kill 90-95% (source: memory/feedback_vivado_serialize_ram.md). Never run Vivado until the design is PROVEN bit-exact AND accurate AND fit-confirmed (source: memory/feedback_vivado_only_when_proven.md). Synth-economy: one synth-only checkpoint + many resume-routes (ONE synth, MANY routes).
- **Weights**: never passed to LLMs — Cartographer writes weights to .hex on disk (output/weights/<id>_weights.hex, <id>_bias.hex); LayerIR carries only metadata + paths; Foundry RTL loads via $readmemh (source: README.md lines 102-124; ARCHITECTURE.md lines 341,352,520). Repacked KPAR8 weight banks (_kp8.mem) are gitignored and must be regenerated on promotion (P0..P4 proofs rerun every invocation) (source: docs/agent_tasks/KPAR8_ANALYSIS.md "PROMOTION CHECKLIST").
- **ImageNet**: the accuracy-measurement calibration (gptq_int4.py, real ImageNet 256/512) differs from the deployment weights (generate_golden synthetic calib) — the default generate_golden does NOT reproduce deployed weights (source: memory/project_deploy_vs_measure_calibration.md, MEMORY index).
- **URAM bitstream init**: UltraRAM CANNOT be non-zero-initialized on U250 / Vivado 2025.2 (UG573 silicon constraint, proven 3 ways: Synth 8-10226 / 8-12183); all weight ROMs live in BRAM/LUT, all URAM is runtime-written (source: memory/project-uram-no-init.md L12-21).
