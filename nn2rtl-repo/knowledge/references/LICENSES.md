# Reference-file provenance

Two reference files are load-bearing in the pattern library today:

## `conv1x1_passing_reference.v`

- **Source**: this repo, generated by Foundry on an earlier run and
  promoted to a reference after it passed end-to-end (sim + Sky130 synth).
  Same license as the rest of this repo.
- **Used by**: `02_conv1x1.md` tells Foundry to adapt this file for any
  pointwise (1×1) conv2d. Proven to produce first-shot passes at
  Fmax 116 MHz on Sky130 for `layer1_0_conv1`.

## `conv3x3_passing_reference.v`

- **Source**: this repo, handwritten from the `03_conv3x3_pad1.md` pattern
  + `rtl_library/coord_scheduler.v` contract. Same license as the rest of
  the repo.
- **Used by**: `03_conv3x3_pad1.md` tells Foundry to adapt this file for
  any 3×3 s1 p1 conv2d.
- **Representative shape**: `layer1_0_conv2` (IC=OC=64, IH=IW=112, MP=4).
  Foundry substitutes channel counts, scale factor, `$readmemh` paths.
- **Structural content**: coord_scheduler instantiation with full
  handshake + combinational `stall_in`, KH-row line buffer with
  right-edge + bottom-edge zero-pad muxing, shift-register window with
  bottom-row data-in bypass, serialized MP-lane MAC loop, window-freeze
  across `OC_PASSES`, round-to-nearest, INT8 saturation, multi-frame
  re-arm on `sched_out_frame_done`.

## Rule for adding future references

Every file in `knowledge/references/` must be actually read by
`foundry.md`'s catalog and actually referenced by a `knowledge/patterns/*.md`
file for adaptation. If neither is true, delete the file — provenance-only
references are noise that mislead both reviewers and the agent.

When we do add a new reference (e.g. a future `conv3x3_passing_reference.v`),
record here: source repo, license, which pattern file consumes it.
